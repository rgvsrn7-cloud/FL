01 — Foundations

Design Philosophy

DataMed transforms from a passive reporting tool into an active decision-intelligence platform. Every design decision serves a single thesis: surface the right action, at the right moment, with the least cognitive friction.

UAE Flag Inspiration — The palette derives from the flag's green (#009A49) and red (#CE1126), recontextualized as semantic healthcare signals: green for revenue health and success; red for denials, risk, and urgency. White fields and neutral dark tones create enterprise professionalism. The result reads as a modern SaaS platform, not a government portal.
Design Principles
01
Action Before Analysis
Every screen's primary job is to tell the user what to do, not merely what happened. KPIs must point toward a next step.
02
Progressive Disclosure
Summary first, detail on demand. Filter panels collapse. Table columns are hidden until invoked. Drawers expand inline.
03
Executive Clarity
Leadership dashboards carry a maximum of 5 primary KPIs. Visual noise is a patient safety risk — clutter obscures critical flags.
04
Workflow First
Denial resolution, resubmission, and reconciliation are guided journeys, not static report pages. Context travels with the user.
05
Data Density Without Clutter
Enterprise users need volume. Tight line-heights, intelligent column defaults, and hover-revealed details keep tables scannable.
06
Healthcare Trust
Color, typography, spacing, and motion choices must convey precision and reliability. No playful gradients on financial data.
07
Operational Visibility
Queue sizes, sync health, and resubmission rates surface in the nav sidebar so operational staff never lose the pulse.
08
Accessibility
WCAG 2.2 AA minimum. 4.5:1 text contrast, 3:1 UI contrast, keyboard-complete nav, focus rings visible on every interactive element.
09
Scalability
Token-driven theming. New facilities, payers, or modules should slot into existing patterns without custom overrides.

05 — Navigation

Single-Sidebar Navigation Framework

DataMed's current dual-nav (top + side) is eliminated. A single persistent left sidebar carries all navigation. The top bar is reserved for context: breadcrumbs, page actions, and user identity.

Decision Rationale — Dual navigation forces users to context-switch between two mental models. A single left rail matches enterprise SaaS conventions (Salesforce, ServiceNow, Workday) and allows the full screen width for data. The sidebar collapses to 60px icon rail on smaller screens.

Navigation Behavior Rules
Badges = Actionable Count
Red badges appear only on items requiring action (denied claims, resubmission queue). Informational counts use neutral badges.
Workspace Switcher
Multi-facility clients switch context via the workspace selector below the logo. The entire sidebar context updates on switch.
Collapsed Rail Mode
At 1280px and below, the sidebar collapses to 60px icon rail. Hover expands on desktop; tap toggles on tablet.
Breadcrumb Hierarchy
Topbar breadcrumb reflects Group / Page / [Record Name]. Maximum 3 levels. Records open in drawers, not full pages.
Global Search (⌘K)
Command-palette search covers claim IDs, member IDs, patient names, and page names. Opens as overlay, not a dedicated page.

Dashboard Layout Rules
KPI Zone — Always First
KPI cards occupy the topmost content row. Maximum 4 on Executive, 6 on Operations dashboards. They summarize the current state and link to detail pages.
Alert Zone — Below KPIs
Priority actions appear immediately below KPIs. These are system-generated based on thresholds: denial spikes, expiring deadlines, sync failures.
Analysis Zone — Below Alerts
Charts, tables, and trend visualizations appear in the lower portion. Users who only need to act can exit at the alert zone without scrolling.

Table Behavior Standards
Sticky Header
Column headers stick at the top of the viewport as users scroll. Essential for tables exceeding one screen height.
Column Management
Users can hide/show columns via a "Columns" gear icon. Default visible columns are curated per module — never show all columns by default.
Saved Views
Filter + column configurations can be saved and shared. "My Denied Claims" or "AXA Pending" views persist across sessions.
Bulk Actions
Checkboxes appear on hover per row. A floating action bar activates when ≥1 row is selected. Actions: Resubmit, Export, Flag, Assign.
Inline Claim Detail
Clicking a Claim ID opens a right-side drawer at 480px width — not a full page navigation. Context is preserved.
Export Behavior
Export applies to the current filtered view, not all data. Export format: XLSX, CSV, PDF. Max 10,000 rows per export. Larger exports are queued and emailed.

Accessibility Requirements Checklist
☑ Keyboard Navigation
All interactive elements reachable and operable by keyboard. Tab order follows visual reading order. Escape closes modals and drawers. Arrow keys navigate dropdowns and tables.
☑ Focus States
Every focusable element has a visible 3px focus ring using brand shadow tokens. Focus rings never removed — outline:none is forbidden without a visible alternative.
☑ ARIA Semantics
Tables use role="table", aria-sort on sortable columns. Status chips use role="status". Alerts use role="alert". Modals use aria-modal="true" and trap focus.
☑ Color-Independent Status
Status chips use color AND icon AND text label. Never rely on color alone. Denied = red chip + "✕" icon + "Denied" label. Assistive technology reads the label.
☑ Screen Reader Support
Data tables have aria-label and a full accessible name. KPI cards announce value + label + delta. Skip-to-content link appears on first Tab press.
☑ Reduced Motion
All animations respect @media (prefers-reduced-motion: reduce). Transitions collapse to instant. Loading states use opacity instead of movement.

Developer Implementation Notes
CSS Custom Properties
All tokens are authored as CSS custom properties on :root. Semantic tokens reference primitive tokens via var(). No hardcoded hex values in component CSS — always use semantic tokens.
Style Dictionary
Use Style Dictionary to transform JSON tokens into CSS, SCSS, and JavaScript for multi-platform output. Configure with a datamed platform config to output --dm-{token} prefixed variables.
Font Loading Strategy
Preload DM Sans 400/700 and Inter 400/500/600 from Google Fonts with display:swap. Subset to Latin + Arabic extended for UAE character support. Cache with service worker for offline use.
Table Virtualization
For tables exceeding 200 visible rows, implement windowed rendering (react-virtual or TanStack Virtual). DataMed claims tables routinely exceed 10,000 rows — DOM performance matters.
